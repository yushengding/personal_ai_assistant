# 执行控制面与自维护规范（v1）

适用范围：`personal_ai_assistant` 的任务执行、可视化监控、用户决策协同、自我升级与空闲自进化。

## 1. 交互原则

- 用户交互入口保持 AIRI 风格：用户只与一个智能体对话下发任务。
- 后台可拆解任务并并发执行，不要求用户理解内部复杂性。
- 所有关键状态在可视化界面实时可见，包含进度、阻塞点、预估完成时间（ETA）。

## 2. 任务模型

## 2.1 实体定义

- `Task`：用户级任务，包含目标、优先级、截止时间、风险级别。
- `SubTask`：由规划器拆解出的最小执行单元，可并发。
- `Checkpoint`：任务执行快照（输入、上下文、输出、中间产物、策略版本）。
- `DecisionTicket`：需要用户决策的问题单，带重要性分级。

## 2.2 状态机

- `Task`：`queued -> planning -> running -> waiting_decision -> completed/failed/cancelled`
- `SubTask`：`ready -> running -> blocked -> retrying -> completed/failed/skipped`
- `Checkpoint`：`created -> verified -> restorable -> archived`

## 3. 并发执行与进度

## 3.1 执行策略

- 规划器输出 `DAG`（有向无环图）而不是线性步骤。
- 调度器基于依赖关系并发执行 `SubTask`。
- 支持资源配额：模型并发上限、工具并发上限、token 预算上限。

## 3.2 进度计算

- `Task Progress = 已完成 SubTask 权重和 / 总权重`
- 权重可由预计耗时、风险级别、用户价值联合计算。
- 进度需要暴露两层：任务总进度 + 子任务泳道进度。

## 4. ETA 预测与历史统计

## 4.1 ETA 预测

- 初始 ETA：基于任务类型模板 + 历史同类任务中位数。
- 运行中 ETA：结合实时耗时、排队时间、重试率、外部依赖延迟动态更新。
- 对每个 `Task`、`SubTask` 都输出：`eta_seconds`, `eta_confidence(0-1)`。

## 4.2 历史统计

- 保存每个历史任务的：`estimated_time`, `actual_time`, `error_rate`, `retry_count`。
- 维护核心统计：
  - 预测误差（MAPE）
  - P50/P90 完成时间
  - 不同任务类型的准时率
- 统计面板必须支持按用户、任务类型、时间窗口筛选。

## 5. 用户 Todo 与决策分级

## 5.1 决策分级

- `L0-Core`（默认最核心）：必须立即用户确认，任务暂停并主动提醒。
- `L1-High`：优先进入待办列表并提示，可短时等待。
- `L2-Normal`：系统可按策略自行决策，保留回退点。
- `L3-Low`：默认自动决策，仅记录日志。

## 5.2 暂停与继续规则

- 遇到 `L0-Core` 的 `DecisionTicket`，`Task` 状态切换到 `waiting_decision`。
- 遇到 `L1-L3`，系统先写 `Checkpoint` 后执行自动决策策略。
- 用户若不满意自动决策结果，可从最近可恢复 `Checkpoint` 一键回退并重跑。

## 5.3 用户 Todo 视图最少字段

- `ticket_id`, `task_id`, `importance_level`, `summary`, `deadline`, `impact`, `recommended_action`
- `requires_pause`（是否阻塞主任务）
- `checkpoint_ref`（可回退快照）

## 6. 可视化大盘（自动完善）

## 6.1 首版固定看板

- 全局运行：活动任务数、并发度、失败率、平均 ETA 偏差。
- 执行拓扑：任务 DAG、关键路径、阻塞节点。
- 资源消耗：token、模型调用、工具调用、队列长度。
- 决策中心：待用户决策票据、分级分布、超时票据。
- 历史绩效：按类型的预计/实际时间对比。

## 6.2 自动完善机制

- 学习引擎周期性分析监控盲点，生成候选指标与告警规则。
- 新增监控项必须经过验证：
  - 信息增益显著（能解释失败/延迟）
  - 不明显增加系统开销
- 通过后自动加入“实验看板”，稳定后升为正式看板。

## 7. 升级不中断与防卡死

## 7.1 安全升级策略

- 双分区运行（`active`/`candidate`），候选版本健康通过后切换流量。
- 长任务升级保护：升级期间禁止中断 `running` 任务，先完成或迁移。
- 心跳与租约：控制平面失联时，执行平面进入保守模式而非停摆。

## 7.2 回滚机制

- 任一升级失败自动回滚至上一个稳定版本。
- 回滚后自动恢复未完成任务到最近 `Checkpoint`。
- 回滚事件强制写入审计并触发告警。

## 8. 空闲自反思与高级组件发明

## 8.1 触发条件

- 系统空闲窗口（CPU/队列/用户交互低于阈值）。
- 学习预算余额充足。
- 当前无高优先级用户任务。

## 8.2 自反思循环

1. 汇总近期失败案例与高成本路径。
2. 生成改进假设（策略、提示、工具组合、流程模板）。
3. 构造离线评测集并回放。
4. 达标后发布为“实验组件/实验策略”。
5. 小流量验证，稳定后正式发布。

## 8.3 发明边界

- 允许：新增策略模板、新监控指标、新执行模板、新工具编排规则。
- 禁止：自动修改核心安全规则、自动放宽权限、自动替换核心执行器。

## 9. 数据表建议（最小）

- `tasks`
- `subtasks`
- `task_checkpoints`
- `decision_tickets`
- `task_time_estimates`
- `task_execution_metrics`
- `upgrade_events`
- `monitor_metric_catalog`

## 10. v1 验收标准

- 并发任务可视化：可实时展示 DAG、进度、ETA、关键路径。
- 历史统计：每个历史任务都有预计/实际时间和误差统计。
- 决策分级：`L0` 必暂停提醒，`L1-L3` 可自动处理并可回退。
- 升级安全：升级失败不丢任务、不失联，5 分钟内完成自动回滚。
- 自我完善：系统可在空闲时提出并验证新监控项或新策略。
