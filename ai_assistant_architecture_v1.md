# AI Assistant 架构目标与落地方案（v1）

## 1. 目标定义

面向 `personal_ai_assistant`，系统必须同时满足：

- 易上手：10 分钟内完成安装、配置、首轮对话。
- 易维护：模块边界清晰，可单测，可替换模型/工具。
- 可自我更新：支持版本化配置、热更新、可回滚。
- 可随使用进化：从用户交互中学习偏好与策略。
- Token 驱动成长：只要有预算，就能持续做离线总结与在线优化。
- 安全与隔离：高风险能力默认隔离、最小权限、可审计。

## 2. 架构原则

- Local-first + Cloud-optional：核心能力本地可运行，外部模型按需接入。
- Gateway-first：所有外部输入/输出统一经过网关，避免旁路。
- Policy-by-default：工具调用与敏感操作必须走策略引擎。
- Learn-with-guardrails：允许学习，但学习过程受预算、权限、验证约束。
- Replaceable components：LLM、记忆库、工具运行时可插拔。

## 3. 建议分层

### 3.1 Interface Layer（接入层）

- `web-client`：聊天、任务、审批、日志可视化。
- `channel-adapters`：后续可扩展 IM/移动端。

### 3.2 Gateway Layer（统一入口）

- 会话路由（session/user/device）
- 请求 schema 校验
- 限流与鉴权
- 事件总线（agent event/tool event/security event）

### 3.3 Agent Runtime（执行层）

- Planner：任务拆解
- Orchestrator：步骤执行与重试
- Tool Caller：工具调用标准化
- Model Router：模型选择（成本/时延/能力）

### 3.4 Memory & Personalization（记忆与个性化）

- 短期记忆：会话窗口 + 摘要压缩
- 长期记忆：偏好、事实、习惯、禁忌
- 用户画像：语气偏好、时间习惯、工具偏好
- 学习管道：交互日志 -> 归纳 -> 候选规则 -> 验证后生效

### 3.5 Learning Engine（成长引擎）

- 在线学习：低风险参数即时更新（如语气、回复长度偏好）
- 离线学习：定时任务生成策略改进建议
- Token 预算器：按日/周预算分配给“服务”与“学习”
- A/B 策略评估：新旧策略小流量对比，达标再全量

### 3.6 Security & Isolation（安全隔离）

- 权限模型：用户级、会话级、工具级
- 工具沙箱：shell/file/web 分级隔离
- 高风险审批：删除文件、外网写操作、系统命令
- 审计日志：谁在何时触发了什么能力，结果如何

### 3.7 Ops & Governance（运维治理）

- 成本看板：token 消耗、学习回报率
- 健康检查：组件状态、延迟、错误率
- 配置中心：版本化配置 + 回滚
- 自更新管理：灰度发布、失败自动回退

## 4. “有 Token 就能成长”的机制设计

成长闭环：

1. 采集：记录交互轨迹（脱敏后）
2. 归纳：提炼用户偏好与常见任务模板
3. 生成：提出可执行的“行为改进规则”
4. 验证：在沙盒/小流量会话中测试规则
5. 发布：策略引擎加载新规则（带版本号）
6. 监控：若成功率下降或风险上升则自动回滚

关键点：

- 学习和服务分预算，避免“越学越贵，主服务变慢”。
- 学习结果不是直接改核心代码，而是先改“策略配置”。
- 只有通过验证门禁（质量、安全、成本）才允许生效。

## 5. 安全与隔离基线（必须先做）

- 默认 deny：未声明权限的工具一律不可调用。
- 双重确认：高风险动作必须人工确认。
- 执行隔离：
  - `safe`：只读检索类工具
  - `guarded`：有副作用但可回滚
  - `restricted`：系统级操作，默认禁用
- 数据隔离：用户记忆按 user_id 分区，禁止跨用户召回。
- 密钥管理：token 仅存储在密钥管理模块，不进入日志。

## 7. 迭代路线图

### Phase 1（可用）

- 网关 + Web 聊天
- 单模型对话
- 基础工具（file/shell/web-search）
- 审计日志 + 人工审批

验收标准：

- 单用户稳定运行 24h
- 所有工具调用可追踪
- 高风险操作无越权

### Phase 2（可靠）

- 记忆系统（短期/长期）
- 成本与延迟看板
- 配置热更新 + 回滚
- 工具分级隔离

验收标准：

- 偏好命中率明显提升
- 出错后 5 分钟内可回滚

### Phase 3（成长）

- 学习引擎上线
- 预算化学习任务调度
- 策略 A/B 与自动发布门禁

验收标准：

- 单周内可观测到响应质量提升
- 学习不影响主对话 SLA

## 8. 关键 KPI

- 用户体验：任务一次完成率、平均交互轮次
- 成本效率：每任务 token 成本、学习投资回报率
- 安全：越权调用次数、高风险误执行次数
- 系统稳定性：P95 延迟、故障恢复时间

## 9. 结论

采用“OpenClaw 的工程骨架 + AIRI 的体验思路”是正确方向。实现顺序应是：

1. 先把安全、隔离、可观测打牢。
2. 再做记忆与个性化。
3. 最后用 token 预算驱动持续成长与自动优化。

这样可以确保系统先可靠，再聪明，最后自进化。
